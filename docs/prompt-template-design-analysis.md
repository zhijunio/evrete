 # 设计分析提示词模板
 
 ## 提示词
 
 你是一名资深 Java 软件架构师，专精于系统设计和代码评审。
 
 ### 任务
 
 分析项目 `/Users/zhijunio/github/evrete` 的设计与实现，系统性地总结其中**做得好的地方**——哪些设计决策、实现手法、架构选择值得称道。将分析结果保存为 `docs/design-analysis.md`。
 
 注意：重点是**好的设计**，不是找问题，也不是面面俱到的文档。
 
 ### 分析框架
 
 请从以下维度展开评审，每个维度给出具体发现：
 
 1. **整体架构与模块划分**
    - 模块的职责边界是否清晰，依赖关系是否合理
    - API / SPI / Runtime 的分离策略
    - 零外部依赖等非功能性决策的代价与收益
 
 2. **API 设计**
    - 类型安全（泛型、Builder 模式、自引用泛型等）
    - 流式 API 的连贯性与可组合性
    - 契约声明（`@NonNull`/`@Nullable`、接口默认方法等）
 
 3. **扩展性与 SPI 机制**
    - Java `ServiceLoader` 的使用方式
    - SPI 接口粒度和抽象层次是否恰当
    - 多提供者优先级排序等机制
 
 4. **核心算法实现（RETE 网络）**
    - 数据结构和算法选择是否贴合领域
    - 两阶段节点分离（知识 vs 会话）的设计意图
    - 图结构抽象的正确性和灵活性
 
 5. **自定义数据结构**
    - `ForkingArray`、`LongKeyMap` 等自建结构的动机
    - 与标准库替代方案（`HashMap`、CopyOnWriteArrayList 等）的取舍
    - 对这些结构的工程复杂度评估
 
 6. **DSL 与注解系统**
    - 注解设计的表达力与限制
    - 动态编译管道的实现路径
    - 字符串字面量处理等精妙细节
 
 7. **运行时能力**
    - 热部署/Delta 更新机制
    - 事件系统与可观测性
    - 线程安全策略
 
 8. **兼容性与适配**
    - JSR94 适配器的厚度与纯定性
    - API 演进中的向后兼容策略
 
 9. **工程实践**
    - 测试覆盖的关键路径
    - 依赖管理理念
    - 代码组织惯例（可见性控制、包结构等）
 
 ### 输出要求
 
 - 保存路径：`docs/design-analysis.md`（覆盖已有文件）
 - 使用中文撰写
 - 每个维度下面至少引用 1 个具体代码示例（包含包名/类名），不要只写空泛的评价
 - 对每个亮点评分（1-5 分），并简短说明评分理由
 - 对于明显**不**好的设计或值得商榷的取舍，也要诚实指出——但要明确标注为"局限/风险"而非"亮点"
 - 在文件末尾增加一个 **Top 5 设计亮点** 的快速榜单
 
 ### 可用的上下文
 
 项目是多模块 Maven 项目，源码位于各模块的 `src/main/java/` 下：
 
 | 模块 | 源码文件数 |
 |---|---|
 | `evrete-core` | ~217 |
 | `evrete-dsl-java` | ~28 |
 | `evrete-jsr94` | ~12 |
 | `evrete-code-samples` | 示例代码 |
 
 **`evrete-core`** 是核心，包含：
 - `org.evrete.api` — 公开 API
 - `org.evrete.api.spi` — SPI 扩展点
 - `org.evrete.runtime` — 运行时代码
 - `org.evrete.runtime.rete` — RETE 网络
 - `org.evrete.collections` — 自建数据结构
 - `org.evrete.spi.minimal` — 默认 SPI 实现
 - `org.evrete.util` — 工具类（含 `StringLiteralEncoder`、`compiler` 等）
 
 全部源码在文件系统中可读，无需外部搜索。
